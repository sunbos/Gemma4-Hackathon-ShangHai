# AshPage 灰页技术报告

## 1. 项目摘要

**AshPage 灰页** 是一个运行在 Android 手机端的离线手写人格陪伴对象。用户在心情不好、想吐槽、想写下某些不适合发给真人的话时，可以直接在手机画布上手写输入；端侧多模态模型在本地理解手写内容，并以指定人格在同一张纸面上留下回应。交互结束后，内容不上传、不沉淀、不形成长期聊天记录，产品体验接近一张会回应、用完即散的“活纸页”。

AshPage 不定位为心理医疗产品，也不试图替代专业咨询。它解决的是更日常、更高频的情绪缓冲需求：用户需要一个低压力、低风险、不评判、不打扰真人的吐槽对象。相比普通聊天机器人，AshPage 的核心差异不是“回答问题”，而是通过手写、纸面批注和人格化回应，提供一个更私密、更有仪式感的情绪出口。

当前 Demo 基于 AshPage 的 Android 端实现，采用原生 Android 画布、LiteRT-LM 本地推理、Agent Skill 和结构化 canvas DSL。模型接收手写画布 PNG，输出包含识别文本、动作类型和纸面元素的 JSON object，Android 端解析后渲染为文本、圈画、箭头、下划线、涂鸦符号等纸面回应。已实现的 Agent Skill 让人格不只是单段 prompt，而是具备可组合的回应策略；下一阶段，AshPage 将重点扩展上下文记忆（Memory）和用户可控的主动打扰能力。

## 2. 产品定位与商业价值

### 2.1 目标用户

AshPage 的目标用户不是“所有需要 AI 聊天的人”，而是更清晰的一类人群：

- 有情绪表达需求，但不想立刻找真人倾诉的人。
- 不方便语音、不想打字聊天、也不想留下聊天记录的人。
- 对隐私敏感，希望情绪内容只在本地短暂存在的人。
- 喜欢手写、日记、角色扮演、虚拟人格陪伴体验的年轻用户。
- 在宿舍、办公室、通勤、深夜等场景中，需要一个短时情绪出口的人。

这些用户的真实需求不是获得标准答案，而是把心里的话写出来，并得到一个“像对象一样”的回应。手写在这里不是低效输入方式，而是产品体验的一部分：它比语音更私密，比打字更慢、更像情绪外化，也更接近日记和纸面倾诉。

### 2.2 真实需求

现实中，用户表达负面情绪有明显成本：

- 找朋友倾诉，担心打扰对方或被评价。
- 发社交平台，担心暴露隐私或留下痕迹。
- 写日记没有回应，情绪很难被接住。
- 使用云端聊天机器人，担心敏感内容上传和历史记录沉淀。

AshPage 的产品价值在于提供一个安全、短暂、可控的表达容器。用户可以把不想说出口的话写下来，由本地人格对象即时回应；内容不上传、不保存，交互结束后像灰烬一样消散。对隐私敏感的情绪场景来说，离线部署不是技术噱头，而是核心商业价值。

### 2.3 商业化潜力

AshPage 的商业化不依赖广告或云端数据沉淀，而更适合围绕本地体验和人格扩展展开：

- **人格包订阅或买断**：不同人格面向不同情绪场景，例如温柔树洞、毒舌朋友、冷静复盘、战略批注、睡前安抚。
- **Agent Skill 扩展**：吐槽、复盘、决策、哄睡、情绪整理、行动建议等能力可以以本地 Skill 形式组合和复用。
- **品牌联名人格**：在合法授权前提下，与内容 IP、校园活动、文创品牌合作推出限定人格。
- **端侧隐私卖点**：不依赖云推理，降低长期服务成本，也能面向隐私敏感用户建立信任。
- **硬件适配空间**：从小米手机扩展到 Android 平板、折叠屏、手写笔设备和其他 Edge AI 终端。

相比依赖云端对话的陪伴类产品，AshPage 的成本结构更轻，隐私叙事更强，也更符合 Edge AI 赛道对离线部署和硬件演示的要求。

## 3. 产品与技术亮点

### 3.1 真实需求与用户价值

AshPage 解决的是明确存在的情绪表达和隐私陪伴需求。它不泛化成心理健康平台，而是聚焦“心情不好时需要一个手写吐槽对象”的轻量场景。这个切入点足够具体，用户路径也足够清晰：

1. 用户打开 App。
2. 在灰色纸页上手写当下情绪或吐槽。
3. 本地模型识别手写内容。
4. 指定人格以纸面批注形式回应。
5. 内容不保存，用户可以清空并离开。

该场景具备真实影响力的原因有三点：

- **隐私强相关**：情绪内容天然敏感，离线处理直接降低用户顾虑。
- **输入方式强相关**：手写比语音和键盘更适合秘密、吐槽和日记式表达。
- **扩展空间明确**：人格包、Agent Skill、纸面风格、Memory 和主动打扰都可以扩展，产品不会停留在单一角色 Demo。

### 3.2 技术实现与工程设计

当前 Demo 采用原生 Android 架构，不使用 WebView，也不依赖本地 HTTP 服务。主要技术链路如下：

- 原生 `InkCanvasView` 捕获用户手写笔迹。
- 画布导出为 PNG bytes，保留纸面输入上下文。
- LiteRT-LM 在本地加载 `.litertlm` 模型。
- 文本和视觉后端使用 GPU，音频后端当前使用 CPU。
- 模型输出严格 JSON object，包含 `recognized_text`、`action` 和 `canvas`。
- Android 端解析 JSON，并映射为原生画布上的纸面元素。

这个架构的优势是端到端链路短，部署边界清晰，隐私风险低。前端画布、模型推理和结果渲染都在手机本地完成，不需要账号系统、云端 API Key 或服务器中转。

在模型调用上，AshPage 不把 Gemma 4 只当作普通文本聊天模型使用，而是让模型承担三类任务：

- **多模态理解**：识别用户手写图像中的内容。
- **人格化生成**：以指定角色语气生成回应。
- **结构化动作生成**：输出可被 Android 执行的 canvas DSL。

当前 Demo 已经实现了 Gemma 4 的原生函数调用式链路，而不是只靠 prompt 工程。`GemmaInference` 在 Android 端通过 `tools = listOf(tool(skillToolSet))` 和 `automaticToolCalling = true` 打开本地 tool calling；`SkillToolSet` 以 `@Tool` 暴露 `loadSkill(name)`，模型在识别到方法论类请求时会主动调用本地 Skill，而不是把完整 Skill 文本预先塞进 prompt。这个设计让模型从“生成文本”升级为“调用本地纸面能力和陪伴策略”，也让函数调用、Skill 路由和端侧执行形成了真实可运行的 Native Function Calling 闭环。

### 3.2.1 模型选型理由

本项目选择 Gemma 4 的 LiteRT-LM 规格，不是因为它“能聊天”，而是因为它同时满足 AshPage 的三个核心约束：

- **端侧可部署**：主要目标设备是小米 13 手机和小米平板 Pro 7，需要在无网环境下本地运行，因此必须优先选择能落在 Android 端、可通过 `.litertlm` 部署的规格。
- **多模态能力**：AshPage 的主输入不是文本，而是手写纸面图像。模型必须同时具备视觉理解和文本生成能力，才能完成“手写识别 + 人格回应 + 结构化纸面动作输出”。
- **Android 生态适配**：当前实现依赖 LiteRT-LM Android 运行时、GPU text/vision backend 和本地 tool calling。Gemma 4 的这一部署路径与现有架构直接兼容，减少了自建推理桥接层的复杂度。

选择这个规格的产品原因也很明确：

- 用户主要使用场景是手机和平板，而不是服务器或桌面端。
- 产品必须覆盖断网、弱网和隐私敏感场景。
- 交互目标是“短而稳的纸面回应”，不是长篇高延迟生成。

因此，本项目没有追求更大但更重的云端规格，而是优先选择适合 Android 端、可落在真实移动硬件上的 Gemma 4 LiteRT-LM 规格。这一选择直接服务于 Edge AI 赛道的离线部署和硬件演示要求。

### 3.3 当前功能完成度

当前 Android 端 Demo 已具备可运行的核心闭环：

- 加载本地 `model.litertlm`。
- 用户在原生纸面画布上手写。
- 支持 3 秒自动提交，也支持手动发送。
- 推理期间显示等待动效。
- 模型返回结构化 JSON 后，App 渲染纸面回应。
- 空白或噪声输入时允许 `stay_silent`，避免强行生成无意义内容。
- 输出解析失败、模型缺失、推理异常时有状态提示。
- 画布元素使用安全锚点布局，避免模型直接输出原始坐标导致越界。

Demo 的核心能力已经覆盖“输入、理解、回应、渲染、异常处理”的完整路径。音频能力在代码中已经存在，但当前提交以手写路径为主，语音能力保留为补充输入方式。

### 3.4 产品创新点

AshPage 的创新点不是“本地 AI 扮演某个角色”，而是把端侧多模态模型变成一个可扩展的手写人格容器。

相比已有“苹果本地 AI + 伏地魔日记”类 Demo，AshPage 的差异化在于：

- 不依赖单一 IP 梗，而是支持多种人格扩展。
- 不显示普通聊天气泡，而是让人格在纸面留下动作和痕迹。
- 不保存完整聊天记录，而是通过“内容阅后即焚 + 本地 Memory 摘要”平衡隐私和可玩性。
- 不把模型输出当作终点，而是把模型输出转换为本地可执行的画布行为。

产品方法上的创新是：把情绪陪伴从聊天框转移到纸面。用户不是向系统提问，而是在和一个人格对象交换情绪；模型不是简单回答，而是在纸上圈画、批注、反问、划掉或留下短句。

## 4. 系统架构

### 4.1 当前实现架构

当前 Demo 采用单端闭环架构：

```text
用户手写
  ↓
Android 原生画布 InkCanvasView
  ↓
导出 PNG bytes
  ↓
Gemma / LiteRT-LM 本地多模态推理
  ↓
结构化 JSON canvas DSL
  ↓
Android 解析 CanvasCommand
  ↓
原生纸面渲染人格回应
```

主要模块职责：

- `MainActivity`：负责页面搭建、模型初始化、按钮交互、自动提交和推理调度。
- `InkCanvasView`：负责手写采集、纸面绘制、等待动效和 canvas command 渲染。
- `GemmaInference`：负责 LiteRT-LM 初始化、发送多模态输入、解析模型 JSON 输出。
- `AudioRecorder`：当前提供音频输入能力，但不是 AshPage 主演示路径。

### 4.2 Canvas DSL

模型输出不是普通自然语言，而是一个结构化对象：

```json
{
  "recognized_text": "今天真的很烦",
  "action": "draw",
  "canvas": [
    {
      "type": "text",
      "text": "那就先别装作没事。",
      "anchor": "below_input",
      "size": "medium",
      "mood": "quiet"
    },
    {
      "type": "underline",
      "target": "input",
      "anchor": "on_input",
      "mood": "sharp"
    }
  ]
}
```

Android 端只接受白名单字段和白名单元素类型，避免模型生成无法执行或越界的 UI 指令。当前支持的元素包括：

- `text`
- `circle`
- `arrow`
- `underline`
- `dot`
- `small_note`
- `spark`
- `bracket`
- `question_mark`
- `speech_bubble`
- `burst`
- `emphasis_lines`
- `scribble`
- `strike`
- `reaction_mark`

这个 DSL 是 AshPage 后续接入原生函数调用的基础。每一种纸面元素都可以被视为一个本地 tool/function，模型负责决定调用哪个动作，Android 负责执行动作。

### 4.3 边缘部署

AshPage 当前使用 `.litertlm` 模型文件进行端侧推理，模型放置路径为：

```text
/sdcard/Android/data/com.fromink.v2/files/model.litertlm
```

部署方式适合 Edge AI 赛道：

- 不依赖云端推理。
- 不需要用户输入上传服务器。
- 小米手机即可完成演示。
- 模型和 App 可随设备携带。
- 弱网、断网和隐私敏感场景下仍可运行。

## 5. Agent Skill、Memory 与主动打扰设计

### 5.1 人格包

AshPage 的人格不是简单提示词，而是一组可组合的产品配置。人格包定义角色气质、回应边界、纸面动作偏好和默认 Skill，使同一个端侧模型可以承载多个不同的陪伴对象：

```json
{
  "id": "ash",
  "name": "Ash",
  "positioning": "毒舌但护短的灰页人格",
  "tone": "短句、克制、直接",
  "boundaries": ["不做医疗诊断", "不鼓励伤害行为", "不保存用户原文"],
  "preferred_canvas_actions": ["small_note", "strike", "underline", "reaction_mark"],
  "default_skills": ["venting", "snap_back", "short_reflection"]
}
```

这种设计能把 AshPage 从单一 Demo 扩展为人格平台。不同人格不仅说话不同，纸面动作也不同：

- 暗色日记人格：低对比墨迹、角落短句、细密批注。
- 温柔树洞人格：柔和小字、少量圈画、更多接纳。
- 战略批注人格：下划线、重点圈画、短促批示。
- 毒舌朋友人格：划掉、感叹号、边角吐槽。

### 5.2 已实现的 Agent Skill

Agent Skill 是 AshPage 已实现的关键能力。它不是模型微调，也不是简单追加 prompt，而是人格可调用的一组本地回应策略。每个 Skill 解决一种具体情绪场景，并决定回应方式、语气强度和纸面动作偏好：

```json
{
  "id": "contradiction_analysis",
  "name": "矛盾分析",
  "trigger_intent": ["反复受挫", "关系冲突", "不知道怎么办"],
  "response_policy": "先接住情绪，再指出主要矛盾，最后给一个很小的行动建议",
  "canvas_style": ["underline", "bracket", "small_note"]
}
```

当前 Agent Skill 的产品价值在于：它把“角色会说什么”拆成可组合的能力模块，使人格可以根据用户输入选择不同策略，而不是每次都给出相似回复。

可扩展 Skill 示例：

- `venting`：陪用户吐槽，不急于解决问题。
- `snap_back`：用更直接的语气把用户骂醒。
- `contradiction_analysis`：拆出主要矛盾和当前阶段任务。
- `sleep_down`：降低刺激，适合睡前情绪。
- `decision_split`：把纠结拆成几个低成本选项。
- `silent_mark`：少说话，只留下一个符号或短句。

Agent Skill 可以通过函数调用式协议表达为本地能力调用：

```json
{
  "function": "use_skill",
  "arguments": {
    "skill_id": "contradiction_analysis",
    "intensity": "medium",
    "canvas_style": ["underline", "small_note"]
  }
}
```

### 5.3 下一阶段：Memory

下一阶段重点不是在手机端持续训练模型，而是实现隐私友好的上下文记忆（Memory）。Memory 的设计原则是：不保存完整原文，不形成可回放聊天记录，只保存本地、可删除、可解释的摘要状态。

```json
{
  "persona_id": "ash",
  "trust_level": 3,
  "reply_length": "short",
  "preferred_skills": ["venting", "contradiction_analysis"],
  "tone_bias": "sharp_but_protective",
  "memory_summary": "用户最近更需要短句接住情绪，而不是长篇建议。",
  "last_active_context": "工作压力"
}
```

Memory 可以带来“人格在变化”的感觉，同时不破坏阅后即焚承诺：

- 用户多次选择短回复，系统减少长段输出。
- 用户经常使用复盘 Skill，系统优先给结构化拆解。
- 信任度提高后，人格语气更直接。
- 用户一键清除状态后，人格回到初始关系。
- 系统记住的是偏好、关系阶段和情绪主题摘要，而不是用户写下的具体内容。

关键点是：系统不保存用户写过什么，只保存本地抽象偏好、关系阶段和可解释摘要。这样既能提升可玩性，也不破坏阅后即焚的产品承诺。

### 5.4 下一阶段：主动打扰

主动打扰不是广告推送，也不是无边界骚扰，而是用户可控的本地陪伴触发机制。它适合解决“用户明明需要情绪出口，但不会主动打开 App”的场景。

可设计为三类：

- **轻触发**：在用户设置的时间段内，灰页以一句很短的话出现，例如“今天要不要写一笔？”
- **关系触发**：当某个人格的信任度提高后，偶尔以更像熟人的方式发起短句。
- **情境触发**：基于本地 Memory 的抽象主题，例如连续几天记录为“工作压力”，主动给出复盘邀请。

主动打扰必须满足三个产品边界：

- 默认关闭，由用户主动开启。
- 触发逻辑本地执行，不上传 Memory。
- 每个人格可单独设置频率、静默时间和关闭开关。

这项能力能明显提高留存和可玩性：AshPage 不再只是用户打开后才回应的工具，而是一个在合适边界内“偶尔想起你”的人格对象。

## 6. 安全、隐私与边界

AshPage 必须明确边界，避免被误解为心理治疗或危机干预工具：

- 不提供心理疾病诊断。
- 不替代专业咨询、医疗或紧急援助。
- 对自伤、伤害他人等高风险表达，应引导用户寻求现实帮助。
- 默认不上传、不保存用户原文。
- 本地人格状态必须可查看、可重置、可删除。
- 角色扮演不能鼓励危险行为、违法行为或现实伤害。

产品文案应避免“治愈”“疗愈疾病”“心理医生”等表述，改用“情绪出口”“吐槽对象”“低压力陪伴”“短时缓冲”等更准确的描述。

## 7. Demo 验收路径

当前 Demo 可按以下路径验证：

1. 将 `model.litertlm` 放入 Android App 外部文件目录。
2. 在小米手机上安装并启动 App。
3. 确认状态从 `Loading model...` 变为 `Ready: model.litertlm`。
4. 在画布上手写一句情绪表达。
5. 等待 3 秒自动提交，或点击发送按钮。
6. 确认等待点出现，随后纸面出现中文人格回应。
7. 观察回应是否包含文本和至少两个纸面标注元素。
8. 清空画布，确认不会保留上一轮内容。

示例输入：

- “今天真的很烦，不想理任何人。”
- “我明明很努力，为什么还是这样？”
- “我只是想有人站在我这边。”

不同人格扩展演示可以使用同一句输入，展示纸面回应风格差异。

## 8. 与竞品和相似 Demo 的差异

2025 年后，已经出现过使用苹果本地 Foundation Models 扮演伏地魔日记的类似创意 Demo。相较于这类展示单一角色新鲜感的方案，本项目的差异化不建立在“本地 AI 角色日记”本身，而建立在可扩展人格、端侧函数调用和结构化纸面渲染的组合能力上。

本项目的差异化主要体现在四点：

| 维度 | 类似角色日记 Demo | AshPage 灰页 |
| --- | --- | --- |
| 产品核心 | 单一角色扮演 | 可扩展人格容器 |
| 交互方式 | 手写后得到回复 | 人格在纸面留下动作 |
| 隐私机制 | 本地模型为主 | 内容阅后即焚，本地 Memory 可控 |
| 可玩性 | 依赖角色新鲜感 | 人格包、Agent Skill、Memory 和主动打扰 |

换句话说，AshPage 不只是“另一个会说话的日记”，而是一套可以承载多种人格、Agent Skill、Memory 和主动陪伴能力的端侧交互框架。

## 9. 下一阶段路线

下一阶段的实现重点如下：

1. **Memory 记忆系统**：保存本地摘要、偏好和关系阶段，不保存用户原文。
2. **主动打扰**：在用户授权下实现本地触发的短句提醒、人格问候和复盘邀请。
3. **阅后即焚动效**：让清空过程变成灰烬、褪色或墨迹消散仪式。
4. **人格包配置化增强**：继续完善本地 JSON 人格包，使人格、Skill、纸面风格可组合。
5. **README 强化**：补充一键构建、模型放置、硬件环境、小米手机演示说明。
6. **演示视频**：录制完整离线流程，明确展示无网络、手写输入、Agent Skill、纸面回应和清空。

这些功能可以逐步上线，不影响当前 Demo 的可运行性。

## 10. 结论

AshPage 灰页的价值不在于再做一个聊天机器人，而在于重新定义端侧多模态模型的交互入口：用户不是打字提问，而是在纸上写下情绪；模型不是输出聊天气泡，而是以人格身份在纸面留下回应；系统不是积累聊天记录，而是让内容短暂存在、随后消散。

从提交材料的产品与工程价值来看，AshPage 同时具备三个优势：

- **真实需求明确**：用户需要隐私友好、低压力的情绪吐槽对象。
- **商业价值清晰**：人格包、Agent Skill、Memory、主动打扰和端侧隐私体验具备扩展空间。
- **技术表达独特**：Gemma 4 多模态理解和结构化纸面动作输出结合，形成区别于普通聊天应用的 Edge AI Demo。

因此，本项目提交的不是一个普通聊天界面，而是一个面向手机和平板场景的端侧多模态人格交互系统。它把离线部署、原生函数调用、手写图像理解、结构化纸面渲染和本地 Skill 扩展组合在同一条 Android 端链路中，既对应真实的隐私与无网使用场景，也体现了 Gemma 4 在 Edge AI 落地中的产品价值和工程价值。
