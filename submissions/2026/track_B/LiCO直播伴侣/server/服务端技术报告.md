# Gemma 4 多模态服务端技术报告

## 1. 项目目标

本服务端用于将原先基于 Qwen 的直播多模态分析能力迁移到本地部署的 Gemma 4 开源模型体系上。服务端提供两类接口：

1. 业务 API：`/api/v1/text/summarize`、`/api/v1/audio/summarize`、`/api/v1/image/analyze`。
2. OpenAI 兼容 API：`/v1/chat/completions`，用于承接客户端原有 Qwen OpenAI 风格请求，包括文本、图片、`input_audio`、流式 SSE、鉴权与模型名映射。

面向的客户端是部署在个人 PC 上的直播监控程序。客户端需要与 OBS、本地音频设备、直播弹幕和本地 UI 低延迟交互，因此保留在个人 PC；Gemma 4 服务端部署在具备 H800 GPU 的远程服务器上，承担大模型推理、ASR、联网搜索和多模态联合分析。

## 2. 模型选型理由

### 2.1 Gemma 4 31B 多模态模型

服务端主模型为 `gemma4-31b-mm`。选择 31B 规格的主要原因是：

- 推理质量：31B 规模更适合直播复杂语境、多模态归因、弹幕统计与画面内容联合判断。
- 视觉能力：31B-mm 与 mmproj 支持图片输入，适合作为视觉场景识别和最终联合推理模型。
- 兼容性：通过 Ollama 的 OpenAI 兼容接口暴露 `/v1/chat/completions`，可以承接客户端原 Qwen 请求结构。
- 可控部署：GGUF/Ollama 部署方式便于在单张 H800 上常驻显存，并通过 `OLLAMA_NUM_PARALLEL=4` 提高并发吞吐。

当前运行环境中，`gemma4-31b-mm:latest` 已常驻 Ollama，显存占用约 21~25GB。服务端启动脚本开启了 `OLLAMA_KEEP_ALIVE=-1`、Flash Attention、KV cache q8_0 和并发槽位 4，以降低冷启动和分轨请求排队时间。

### 2.2 Gemma 4 E4B 多模态音频模型

服务端保留 `gemma4-e4b-mm` 作为音频模型配置。E4B 的定位是原生音频/轻量多模态理解模型，适合较轻量、低成本的音频场景。但实际测试中，31B 原生音频支持不足，而 E4B 的 ASR 转写稳定性不满足直播状态分析要求，因此最终架构采用：

- 31B 负责最终文本、视觉、联合推理。
- faster-whisper 作为外部 ASR 工具，先转写音频，再把 ASR 结果注入 31B 上下文。
- E4B 保留为兼容旧接口和后续原生音频能力扩展的候选模型。

### 2.3 外部 ASR 工具的必要性

直播音频分析要求不仅要判断是否有人声，还要区分主播语音、桌面音频、BGM、游戏声和静音。Gemma 4 31B-mm 没有稳定的原生音频编码能力，因此服务端引入 faster-whisper：

- 使用本地 `/root/Gemma/whisper-base` 模型，避免在线下载不稳定。
- 使用 CUDA/float16 运行在 H800 上。
- ASR 前保存客户端原始音频，使用 ffprobe/ffmpeg 进行时长、采样率、峰值、RMS、静音判断。
- 若 VAD 过滤后无结果，会关闭 VAD 重试，降低漏检低音量人声的概率。

## 3. 架构设计

### 3.1 客户端与服务端分离部署

客户端部署在个人 PC，服务端部署在 H800 服务器。这样设计是因为：

- 客户端必须低延迟接入 OBS，读取当前画面、麦克风轨、桌面音频轨和弹幕流。
- OBS、音频设备、窗口截图和直播 UI 都位于个人 PC，本地采集可以减少端到端延迟和复杂的远程设备访问问题。
- 服务端需要大显存 GPU，适合部署在 AutoDL/服务器环境，专注模型推理和工具执行。
- 二者通过 HTTP/OpenAI 兼容协议通信，可通过 AutoDL 自定义服务 URL 或 SSH 隧道连接；实际压测表明，`:8443` 网关上传带宽可能成为瓶颈，推荐 SSH 隧道直连 6006。

### 3.2 服务端进程与组件

核心进程如下：

- FastAPI/Uvicorn：监听 `0.0.0.0:6006`，提供业务 API 与 OpenAI 兼容 API。
- Ollama：监听 `127.0.0.1:11434`，加载 `gemma4-31b-mm` 和可选 `gemma4-e4b-mm`。
- faster-whisper：作为 Python 进程内缓存模型，用于 ASR 工具。
- Playwright/browser-use：用于 web_search 后的深度网页阅读。
- 百度千帆 AI Search：用于中文互联网事实检索。

### 3.3 多模态请求包含的信息

直播联合请求通常包含：

- 文本任务指令：要求输出严格 JSON，包含心理分析、语音报告、画面场景报告。
- 弹幕/统计 JSON：弹幕数量、速率、样例消息、礼物、压力词、直播环境提示等。
- 图片：OBS 实时截图，压缩为 JPEG 后以 `image_url.url=data:image/jpeg;base64,...` 传入。
- 音频 1：麦克风轨，`input_audio`。
- 音频 2：桌面音频轨，`input_audio`。
- 服务端派生上下文：ASR 转写、音频诊断、静音判断、文件路径、请求 ID。

服务端限制单次联合请求最多 2 段音频、2 张图片、单段音频默认不超过 8 秒。输出 token 上限和主推理短超时已取消，由客户端/评测端决定等待策略。

## 4. 多模态处理流程

### 4.1 OpenAI 兼容入口

`gemma_api/routers/openai_compat.py` 是最关键的兼容层。它完成：

1. Bearer API Key 鉴权。
2. Qwen/Gemma 模型名映射。
3. 保存客户端原始请求体到 `received_requests/`，便于复现。
4. 解析 `input_audio` 和 `image_url`。
5. 保存音频到 `received_audio/`。
6. 执行音频诊断和 ASR。
7. 将 `input_audio` 替换为文本提示，并注入 `[服务端音频分析上下文]`。
8. 将图片、文本、弹幕统计、ASR 结果统一交给 `gemma4-31b-mm` 做单次联合推理。
9. 返回 OpenAI 兼容 SSE 或非流式 JSON。

### 4.2 音频处理

音频处理分为三层：

- 原始文件保存：保留客户端传入的真实音频用于审计和复现。
- 诊断：ffprobe 读取格式/时长，ffmpeg 解码为 16kHz PCM 后计算 peak、RMS、dBFS 和 likely_silent。
- ASR：faster-whisper 转写中文，先用 VAD，再在空结果时关闭 VAD 重试。

ffmpeg/ffprobe 是阻塞进程，因此在 async handler 中用 `asyncio.to_thread` 执行，避免阻塞 Uvicorn 事件循环。

### 4.3 图片处理

OpenAI 兼容入口透传 `image_url` 到 Ollama/Gemma 31B-mm。客户端应把 OBS 截图压缩为 JPEG、最长边约 1280px，以减少上传时间。服务端保存完整原始请求体，但不单独落盘图片；必要时可从 `received_requests/*.json` 中提取。

### 4.4 SSE 与 reasoning/content 兼容

Ollama/Gemma 在部分情况下会把模型思考过程放到 `delta.reasoning`，而 `delta.content` 为空。该 reasoning 可能是英文，不应作为业务报告展示。服务端已经：

- 强化 system prompt，要求最终输出简体中文 JSON。
- 不再把 reasoning 原样暴露为 content。
- 若模型只有 reasoning 没有最终 content，返回中文降级 JSON。

前端应只展示 `delta.content`，`delta.reasoning` 只允许进入 debug 日志。

## 5. Gemma 4 31B 使用的 Tools

### 5.1 web_search

用途：当内部知识不足、问题涉及热点、网络梗、最新事实或高风险事实核验时联网检索。

实现：`gemma_api/services/search.py`。当前搜索后端优先为百度千帆 AI Search（`baidu_ai`），也保留 Sogou、DuckDuckGo、HTTP 抓取与 browser-use 深读能力。

关键策略：

- 搜索 query 不自动扩展，避免改变用户问题。
- 高风险事实问题必须结合当前时间与检索证据。
- 无搜索结果时明确返回“未找到相关搜索结果”。
- 深读 TopN 页面并输出进度。

### 5.2 get_current_time

用途：为“现在/最新/是否去世/是否在世”等时效性问题提供服务器真实时间锚点。

实现：`gemma_api/services/agent.py` 中工具声明与 `_current_time_payload()`。

### 5.3 asr_transcribe

用途：对上传音频执行高精度 ASR，弥补 31B-mm 原生音频能力不足。

实现：`gemma_api/services/stt.py` 与 OpenAI 兼容层的强制 ASR。联合推理模式下不再等待模型自主决定调用工具，而是服务端预先执行 ASR 并将结果注入上下文。

### 5.4 browser-use / Playwright 深读

用途：对 web_search 的 TopN 网页进行正文阅读，补充搜索摘要不足的问题。

实现：`gemma_api/services/deep_read.py` 与 `search.py` 中 browser-use 集成。当前启动脚本把 `BROWSER_USE_MAX_STEPS` 控制在 12，并降低 action wait/idle wait 以减少延迟。

### 5.5 echo_size 调试工具

用途：只读取请求体并返回字节数与服务端读取耗时，用于区分“上传慢”和“推理慢”。

实现：`POST /v1/echo_size`。该工具不参与模型推理，但对部署诊断非常重要。

## 6. 性能与稳定性设计

- 预热：FastAPI startup 时预热 Ollama 主模型和 Whisper，降低首请求冷启动。
- 并发：`OLLAMA_NUM_PARALLEL=4` 支持分轨音频、视觉和联合请求并发。
- 显存：H800 80GB，31B-mm 常驻约 22GB，Whisper 约 0.7GB。
- ASR 并发控制：`ASR_MAX_CONCURRENCY=1`，避免 CTranslate2/Whisper 多路争抢 GPU 导致超时。
- 请求落盘：保存原始请求与音频，支持 replay 和故障复现。
- SSE 心跳：流式响应先发送 `: connected`，等待模型首 token 期间每 10 秒发送 `: keepalive`。
- 上传诊断：`echo_size` 用于确认 AutoDL 网关或 SSH 隧道上传性能。

## 7. 代码目录说明

- `gemma_api/main.py`：FastAPI 应用入口、CORS、路由注册、模型预热、健康检查。
- `gemma_api/config.py`：模型名、端口、API Key、搜索、Whisper、browser-use、路径配置。
- `gemma_api/routers/openai_compat.py`：OpenAI 兼容层、多模态联合推理、ASR 强制执行、请求保存、SSE 兼容、echo_size。
- `gemma_api/routers/text.py`：文本总结业务 API。
- `gemma_api/routers/audio.py`：音频总结业务 API，支持 E4B 原生音频与 Whisper fallback。
- `gemma_api/routers/image.py`：图片分析业务 API。
- `gemma_api/services/agent.py`：Gemma Agent、工具声明、搜索/时间工具调用、事实核验规则。
- `gemma_api/services/search.py`：百度千帆 AI Search、传统搜索、browser-use 深读。
- `gemma_api/services/stt.py`：faster-whisper ASR 与音频诊断。
- `gemma_api/services/ollama_client.py`：Ollama API 封装。
- `start_gemma_api.sh`：一键后台启动脚本。
- `scripts/setup_gemma311_cn.sh`：国内镜像安装 Python 3.11、依赖与 Playwright。
- `replay_request.py` / `stress_replay.py` / `profile_latency.py`：调试、回放、压测工具。

## 8. 当前测试硬件环境

- GPU：NVIDIA H800 PCIe，显存 81,559 MiB。
- Driver：580.82.07。
- CUDA：13.0。
- 操作系统：Linux 5.15.0-161-generic。
- Python 运行环境：Conda `gemma311`，Python 3.11。
- 推理引擎：Ollama 0.30.4。
- 常驻模型：`gemma4-31b-mm:latest`，OpenAI 兼容服务端映射名 `gemma4-31b-mm`。
- 服务端端口：6006。
- Ollama 端口：11434。

## 9. 结论

本服务端采用“远程 H800 大模型服务 + 本地 PC OBS 客户端”的分布式架构，在不改动客户端 OpenAI 请求主体的前提下，完成 Qwen 多模态能力向 Gemma 4 的迁移。31B-mm 作为高质量视觉/文本/联合推理主模型，E4B 作为音频模型兼容项，faster-whisper 作为可靠 ASR 工具，百度 AI Search/browser-use 作为联网事实增强工具。该架构兼顾了直播端低延迟采集、服务端高算力推理、OpenAI 兼容性、可复现调试和工程可部署性。
