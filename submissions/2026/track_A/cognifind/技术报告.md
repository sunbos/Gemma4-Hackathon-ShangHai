# CogniFind 技术报告

## Gemma 4 模型选型与架构设计

---

## 1. 项目概述

CogniFind 是一款跨平台桌面端智能文件搜索与 AI 助手应用，核心技术栈为 **Rust**（高性能桌面应用）+ **Gemma 4**（大语言模型推理）。项目采用"本地文件索引 + AI 理解"的双引擎架构，通过 RAG（检索增强生成）技术实现对用户本地文件的智能问答。

- **赛道**: A — AI Agent（原生函数调用 + 多步规划工具链）
- **项目名称**: CogniFind
- **核心模型**: Google Gemma 4 E2B（Edge 2B，端侧本地模型）

## 2. 模型选型理由

### 2.1 为何选择 Gemma 4 家族

本项目在模型选型时对标了以下候选方案：

| 维度 | Gemma 4 E2B | GPT-4o | Llama 4 | Qwen3 |
|------|:---:|:---:|:---:|:---:|
| **原生函数调用** | 专用 Token 体系 | JSON Mode | 无原生支持 | 部分支持 |
| **多模态** | 原生 image/audio/video | 支持 | 部分支持 | 部分支持 |
| **开源部署** | 完全开源 | 闭源 | 开源 | 开源 |
| **端侧部署** | 2B 参数，CPU 可运行 | 不可 | 困难 | 困难 |
| **隐私保护** | 完全本地，数据不出设备 | 需上传云端 | 本地可部署 | 本地可部署 |
| **中文支持** | 优秀 | 优秀 | 一般 | 优秀 |

#### 关键决策因素：

**（1）原生函数调用（Native Function Calling）—— 核心差异化能力**

Gemma 4 是目前唯一提供 **原生 Token 级函数调用** 的模型，通过 6 个专用特殊 Token 实现：

- `<|tool|>` / `<|tool|>` — 工具定义边界
- `<|tool_call|>` / `<|tool_call|>` — 模型请求调用工具
- `<|tool_result|>` / `<|tool_result|>` — 工具执行结果返回

这种设计相比 GPT-4o 的 JSON Mode 有本质优势：
- Token 级精确边界，避免 JSON 解析错误
- 多轮工具调用中状态管理更可靠
- 不需要复杂的 prompt 工程来约束输出格式

对于 CogniFind 的 **混合路由 Agent 架构**（五维决策法），原生函数调用是实现文件搜索、内容索引、代码分析等多工具协同的基础。

**（2）端侧部署的极致效率**

Gemma 4 E2B 仅 2B 参数，专为边缘设备与端侧场景设计，意味着：
- **纯 CPU 即可运行**，无需 GPU，无需 CUDA 环境
- 普通笔记本（8GB RAM）即可流畅推理
- 模型文件仅约 4GB，下载和加载极快
- 推理速度在 CPU 上可达 20-30 tokens/s

**（3）完全本地化，隐私零泄露**

CogniFind 处理的是用户本地文件数据，隐私保护是核心需求。E2B 作为本地模型：
- 所有推理在用户设备上完成，数据永不离开本地
- 无需 API 密钥，无需网络连接，支持离线使用
- 符合 GDPR、个人信息保护法等合规要求

**（4）多模态融合能力**

Gemma 4 E2B 同样原生支持 `<|image|>`、`<|audio|>`、`<|video|>` 三种模态输入，满足 CogniFind 对文件缩略图理解、PDF 截图 OCR、音频笔记转写等场景需求。

**（5）完全开源 + 商业友好许可**

Gemma 4 使用 Apache 2.0 许可，允许商业使用，无任何使用限制。

### 2.2 为何选择 E2B（2B 端侧）规格（而非 26B MoE / 31B Dense）

| 规格 | 能力 | 部署要求 | 适用场景 |
|------|------|------|------|
| **E2B (2B)** | **原生函数调用 + 多模态 + 端侧推理** | **CPU / 8GB RAM** | **桌面端本地 AI Agent** |
| 4B | 基础推理 | 移动端 / 轻量 GPU | 轻量 agent |
| 26B MoE | 强推理 + 多模态 | 消费级 GPU (24GB) | 服务端部署 |
| 31B Dense | 最强推理 | 服务端 GPU (A100) | 云端部署 |

选择 E2B 的核心逻辑：

1. **隐私优先的设计哲学**：CogniFind 的核心价值主张是"本地文件智能搜索"，用户数据隐私是不可妥协的底线。E2B 完全本地运行，无需上传任何数据到云端，这是 26B MoE 等依赖 GPU 服务端部署方案无法满足的。

2. **零部署门槛**：CogniFind 面向普通桌面用户，E2B 仅需 8GB RAM + CPU 即可运行，无需 GPU、无需 CUDA、无需 Docker。用户可以"开箱即用"，符合消费级桌面应用的体验标准。

3. **原生函数调用能力不缩水**：E2B 虽然参数规模小，但完整保留了 Gemma 4 的原生函数调用 Token 体系（`<|tool|>`、`<|tool_call|>`、`<|tool_result|>`），Agent 工具链能力与大型号完全一致。

4. **多模态能力完整保留**：E2B 同样支持 `<|image|>`、`<|audio|>`、`<|video|>` 三种模态输入，满足文件搜索场景下的多模态需求。

5. **混合路由架构弥补能力差距**：对于 E2B 处理不了的复杂推理任务（如深度代码分析、数学证明），CogniFind 的五维混合决策引擎会自动路由到外部大模型（云端 API），实现"本地快 + 云端强"的互补。

### 3.2 核心模块设计

#### 3.2.1 推理路由引擎（engine.rs）

**设计目标**：在本地模型与外接 Gemma 4 API 之间智能路由，兼顾隐私保护与推理能力。

**五维混合决策法**：

```rust
fn decide_hybrid_backend(cfg, ctx, question) -> Decision {
    // 维度 1: 数据敏感度 — 含本地文件上下文 → 本地
    if has_rag_context(ctx) { return Local; }
    // 维度 2: 任务复杂度 — 代码/数学/推理 → 外部
    if is_complex_task(question) { return External; }
    // 维度 3: 延迟要求 — 简单任务 → 本地
    // 维度 4: 本地资源 — 本地模型就绪 → 本地
    // 维度 5: 网络状态 — 外部 API 就绪 → 外部
    // 优雅回退：目标后端不可用时自动切换
}
```

**关键设计**：
- 原子取消标志（`AtomicBool`）支持用户中途停止推理
- `catch_unwind` 异常隔离，防止推理线程崩溃影响主进程
- 后端不可用时自动优雅回退（Local → External 或 External → Local）

#### 3.2.2 外接 API 调用（external_llm.rs）

通过 OpenAI 兼容 Chat Completions API 调用 Gemma 4，支持：

- **纯文本流式问答**：`stream_external_completion()` — SSE 流式接收 token
- **多模态 Vision**：`stream_vision_api()` — 发送 `ContentPart { type: "image_url", ... }` 结构
- **图片描述**：`describe_image_sync()` — 同步获取图片理解结果
- **音频转写**：通过 Whisper 兼容 API 联动

#### 3.2.3 RAG Prompt 构建（rag.rs）

`build_prompt()` 将本地文件检索结果拼接为结构化 prompt：

```
以下是检索到的本地文件上下文，请结合上下文回答用户问题。
若上下文不足请说明。

用户搜索词: {query}

相关文件:
1. report.pdf — /home/user/docs/report.pdf
2. slides.pptx — /home/user/presentations/slides.pptx

内容片段:
1. [/home/user/docs/report.pdf] 本季度营收增长 23%...
2. [/home/user/presentations/slides.pptx] 技术方案包括三个核心模块...

用户问题: {question}
```

#### 3.2.4 AI 桥接服务层（ai_bridge.rs）

封装推理生命周期管理：
- `submit()` — 提交推理任务，返回流式事件通道
- `stop()` — 取消推理
- `drain_events()` — 流式 token 消费

## 4. Gemma 4 原生函数调用实现

### 4.1 原生 Token 机制

Gemma 4 的函数调用不同于传统 JSON Mode 方案，它使用 **词汇表级别的专用 Token**：

```
用户消息 → [系统提示 + 工具定义]
           ↓
Gemma 4 推理 → 判断是否需要调用工具
           ↓
    需要 → 生成 <|tool_call|> 函数名 + 参数 <|tool_call|>
           ↓
    执行工具 → 返回 <|tool_result|> 结果 <|tool_result|>
           ↓
    模型继续推理 → 生成最终自然语言回答
```

### 4.2 CogniFind 中的 Agent 工具链

在 `gemma4_client.py` 中通过 `chat_with_tools()` 实现完整的 Agent 循环：

```python
def chat_with_tools(self, messages, max_tool_rounds=5) -> str:
    """
    1. 发送用户消息 + 工具定义 → Gemma 4 分析
    2. 模型判断是否需要调用工具 → 生成 tool_calls
    3. 执行工具 → 返回 tool_result
    4. 模型生成最终自然语言回答
    """
    for _ in range(max_tool_rounds):
        response = self.chat(current_messages)
        tool_calls = message.get("tool_calls")
        if not tool_calls:
            return message.get("content", "")
        # 执行工具并返回结果
        for tc in tool_calls:
            result = self._execute_tool(tc["function"]["name"], tc["function"]["arguments"])
            current_messages.append({"role": "tool", "tool_call_id": tc["id"], "content": result})
```

**CogniFind 中注册的工具函数示例**：

| 工具名称 | 功能 | 对应 Gemma 4 Token |
|---------|------|-------------------|
| `search_files` | 搜索本地文件 | `<|tool_call|>` 触发 |
| `read_file_content` | 读取文件内容 | 同上 |
| `get_file_metadata` | 获取文件元信息 | 同上 |
| `analyze_code` | 代码分析 | 同上 |

## 5. 多模态处理实现

### 5.1 Gemma 4 多模态 Token 体系

Gemma 4 原生支持三种模态输入，通过专用 Token 实现：

| Token | 模态 | CogniFind 应用场景 |
|-------|------|-------------------|
| `<|image|>` | 图像 | 图片文件内容理解、PDF 截图 OCR |
| `<|audio|>` | 音频 | 音频笔记转写、会议录音分析 |
| `<|video|>` | 视频帧 | 视频文件关键帧识别 |

### 5.2 Vision API 调用链路

```
用户选择图片文件
    ↓
external_llm.rs: image_to_data_uri()
    → 读取文件 → Base64 编码 → 构造 data:image/{mime};base64,{data}
    ↓
external_llm.rs: stream_vision_api()
    → 构建 ContentPart { type: "image_url", image_url: { url, detail } }
    → POST /v1/chat/completions
    → SSE 流式接收 token
    ↓
Gemma 4 处理 <|image|> token → 理解图片内容 → 生成文本回答
```

### 5.3 多模态消息构建

```python
def build_multimodal_message(self, role, content) -> Dict:
    """
    构建多模态消息 — 支持文本 + 图片 + 音频混合输入
    """
    parts = []
    if content.text:
        parts.append({"type": "text", "text": content.text})
    if content.image_path:
        parts.append({"type": "image_url", "image_url": {"url": self._encode_image(content.image_path)}})
    if content.audio_path:
        parts.append({"type": "audio_url", "audio_url": {"url": self._encode_audio(content.audio_path)}})
    return {"role": role, "content": parts}
```

## 6. 技术亮点总结

1. **原生函数调用（Native Function Calling）**：利用 Gemma 4 专用 Token 体系实现 Agent 工具链，支持多轮工具调用与结果回传，优于传统 JSON Mode 方案。

2. **混合路由引擎**：五维决策法在本地隐私与外部能力之间智能平衡，本地模型处理敏感文件数据，外接 Gemma 4 处理复杂推理任务。

3. **多模态融合**：原生支持图片、音频、视频三种模态，满足文件内容理解的全场景需求。

4. **RAG 增强**：本地文件索引 + AI 理解的双引擎架构，实现"搜索即对话"的智能文件交互体验。

5. **跨平台桌面应用**：基于 Rust + egui 构建，支持 Windows/macOS/Linux，通过全局热键随时唤出 AI 助手。
